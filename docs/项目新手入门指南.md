# 项目新手入门指南

> 这份文档给第一次接触 `mcp-manager` 的同学看。
> 
> 如果你已经会用 Go，但还不熟这个项目，建议先看这里；想查完整接口、部署与配置细节，再回到根目录 `README.md`。

## 1. 先用一句话认识这个项目

`mcp-manager` 是一个 **MCP 服务管理平台**：

- 它负责保存 MCP 服务配置
- 负责连接 / 断开 MCP 服务
- 负责同步工具列表
- 负责调用工具
- 负责记录调用历史和审计日志
- 还支持单体部署与 control-plane / executor 双角色部署

你可以把它先理解成：

```text
它不是具体的 MCP 服务本身，
而是“管理很多 MCP 服务，并替你统一调用它们”的后台系统。
```

## 3. 第一次启动：建议走最轻量路径

虽然项目默认配置更偏向 **PostgreSQL + Redis**，但如果你只是本地学习代码，建议先走最简单路径：

```bash
MCP_DATABASE_DRIVER=sqlite \
MCP_DATABASE_DSN=data/mcp_manager.db \
MCP_REDIS_ENABLED=false \
go run ./cmd/server
```

为什么推荐这样做？

- 不需要先准备 PostgreSQL
- 不需要先准备 Redis
- 更适合新手先把服务跑起来
- 等你理解项目后，再切回默认的 PostgreSQL / Redis 部署

启动后可以先访问：

```text
http://127.0.0.1:8080/health
http://127.0.0.1:8080/ready
http://127.0.0.1:8080/swagger/index.html
```

默认管理员账号：

- 用户名：`root`
- 密码：`admin123456`

## 4. 项目是怎么启动起来的

新手最值得先看的不是某个 handler，而是启动主链路。

### 4.1 启动主链路

对应文件：`cmd/server/main.go`

```text
main
 ├─ config.Load(".")              读取配置
 ├─ logger.Init(...)              初始化日志
 ├─ buildApp(cfg)                 装配整个应用
 │   └─ bootstrap.NewBuilder(cfg).Build()
 ├─ serveApp(...)                 启动 HTTP 服务
 └─ shutdown / cleanup            优雅退出与资源清理
```

### 4.2 为什么 `internal/bootstrap/app.go` 很重要

这个文件可以理解成项目的“总装车间”。

它会把这些东西拼起来：

- 数据库初始化与迁移
- 默认管理员初始化
- Redis 客户端
- JWT 服务
- Repository
- Service
- Gin Router
- 健康检查任务
- 审计清理任务
- Executor RPC Server（特定角色下）

你可以把它理解成：

```text
main.go 只负责“叫人干活”
bootstrap/app.go 负责“真的把所有零件装起来”
```

### 4.3 装配顺序图

```text
config
  |
  v
bootstrap.Builder.Build()
  |
  +--> database.Init
  +--> database.Migrate              (control-plane / all)
  +--> EnsureAdmin                   (control-plane / all)
  +--> repository.NewXXXRepository
  +--> service.NewXXXService
  +--> health checker                (all / executor 且启用时)
  +--> cleanup task                  (control-plane / all)
  +--> router.New
  +--> RPC server                    (executor 且启用 RPC)
  |
  v
返回 App{Engine, Server, RPCServer, Cleanup}
```

```text

│   ├── service                核心业务逻辑
│   ├── mcpclient              管理 MCP 连接与调用
```

### 可以这样理解每层职责

```text

router       = “把 URL 映射到 handler，并挂中间件”

```

## 6. 读代码时，建议按这个顺序

如果你一上来就钻进 `service` 细节，很容易迷路。

更推荐下面这个顺序：

### 第一轮：先看骨架

1. `cmd/server/main.go`
2. `internal/bootstrap/app.go`
3. `internal/router/router.go`
4. `internal/config/config.go`
5. `config.yaml`

这一轮只回答 3 个问题：

- 程序怎么启动？
- 有哪些依赖？
- 对外暴露哪些路由？

### 第二轮：再看一条完整业务链

推荐看“服务管理”这条链：

1. `internal/handler/mcp_handler.go`
2. `internal/service/mcp_service.go`
3. `internal/repository/mcp_service_repository.go`
4. `internal/domain/entity/mcp_service.go`
5. `internal/mcpclient/manager.go`

这条链最能帮你理解项目的主价值。

### 第三轮：看测试，理解真实用法

推荐看：

1. `tests/testutil/app.go`
2. `tests/integration`
3. `tests/e2e`

因为测试文件往往比业务代码更像“使用说明书”。

## 7. 一条请求是怎么流转的

这里用“创建并连接一个 MCP 服务”举例。

### 7.1 从 HTTP 到数据库/运行时

```text
客户端 POST /api/v1/services
        |
        v
router/router.go
        |
        v
handler/mcp_handler.go
  - 绑定请求体
  - 从中间件里取当前用户
        |
        v
service/mcp_service.go
  - 校验输入
  - 构造 entity.MCPService
  - 记录审计
        |
        v
repository/mcp_service_repository.go
  - 用 GORM 写入数据库
```

### 7.2 连接 MCP 服务时会多一步运行时交互

```text
客户端 POST /api/v1/services/:id/connect
        |
        v
handler/mcp_handler.go
        |
        v
service/mcp_service.go::Connect
        |
        +--> repository.UpdateStatus(CONNECTING)
        |
        +--> connector.Connect(...)
               |
               +--> 本地模式: local runtime adapter -> mcpclient.Manager
               |
               +--> 双角色模式: remote runtime adapter -> internal/rpc
        |
        +--> repository.UpdateStatus(CONNECTED / ERROR)
        +--> audit / alert
```

这就是本项目最关键的地方：

```text
“配置和审计”在数据库侧
“真实连接和调用”在 runtime 侧
service 层负责把两边串起来
```

## 8. 为什么会有 all / control-plane / executor 三种角色

这个项目不是只有一种运行方式。

### 8.1 三种角色的直觉理解

| 角色 | 直觉理解 | 适合什么场景 |
| --- | --- | --- |
| `all` | 一个进程把管理和执行都做了 | 本地开发、简单部署 |
| `control-plane` | 只负责管理接口和数据库 | 想把执行面拆出去 |
| `executor` | 只负责真正连接/调用 MCP 服务 | 双角色部署中的执行节点 |

### 8.2 角色关系图

```text
             +---------------------------+
             |       control-plane       |
             | API / DB / 审计 / 管理逻辑 |
             +-------------+-------------+
                           |
                           | internal RPC
                           v
             +---------------------------+
             |         executor          |
             | 真实连接 MCP / 调工具执行 |
             +---------------------------+

all = 上面两者合在同一个进程里
```

### 8.3 为什么 `executor` 模式下看不到业务 API

因为 `internal/router/router.go` 里明确写了：

```text
如果 role == executor：
只注册 /health /ready /swagger
不注册 /api/v1/... 业务接口
```

所以如果你在 `executor` 模式下访问 `/api/v1/services` 得到 404，这不是 bug，而是设计如此。

### 8.4 不要把“应用角色”和“用户权限角色”混在一起

前面说的 `all / control-plane / executor`，是**进程怎么部署**。

但系统里还有另一套 role，是**登录用户的权限角色**：

| 用户角色 | 含义 | 代码中的权限效果 |
| --- | --- | --- |
| `admin` | 管理员 | 可访问管理接口，也可修改 |
| `operator` | 操作员 | 可修改服务和工具，但不能做管理员操作 |
| `readonly` | 只读用户 | 只能看，不能改 |

可以用下面这张图记忆：

```text
应用角色（进程怎么跑）
  all / control-plane / executor

用户角色（登录后能做什么）
  admin / operator / readonly
```

如果你在 Swagger 里看到 403，优先检查的是**当前登录用户角色**；如果你看到 `/api/v1/...` 直接 404，再检查的是**当前应用运行角色**。

## 9. 配置系统要怎么理解

配置入口在：`internal/config/config.go`

### 9.1 配置优先级

```text
MCP_* 环境变量
   > config.yaml
   > 代码里的默认值
```

### 9.2 新手最需要知道的几个配置

| 配置 | 作用 | 新手建议 |
| --- | --- | --- |
| `app.role` | 决定运行角色 | 先用 `all` |
| `database.driver` | 数据库驱动 | 本地学习先用 `sqlite` |
| `database.dsn` | 数据库连接串 | SQLite 可直接指向本地文件 |
| `redis.enabled` | 是否启用 Redis | 本地学习可先关掉 |
| `rpc.enabled` | 是否启用双角色 RPC | `all` 模式可关闭，dual-role 必须开启 |
| `jwt.secret` | JWT 签名密钥 | 开发环境可用默认，正式环境必须改 |

### 9.3 新手最常见误区

#### 误区 1：直接 `go run ./cmd/server` 就应该能跑

不一定。

因为当前默认配置更偏向：

- PostgreSQL 已就绪
- Redis 已就绪

如果你的本地没有这两个服务，就应该显式覆盖成 SQLite + 关闭 Redis。

#### 误区 2：`config.yaml` 改了，环境变量也会跟着改

不会。

环境变量优先级更高。如果行为和 `config.yaml` 不一致，先检查有没有设置过 `MCP_*` 变量。

#### 误区 3：dual-role 时可以像 all 模式一样把 RPC 关掉

不行。

当前代码里 `control-plane` 和 `executor` 两种模式都要求 `rpc.enabled=true`，否则配置加载后的校验阶段就会报错。

## 10. 这个项目里的“核心业务对象”有哪些

### 10.1 `MCPService`

文件：`internal/domain/entity/mcp_service.go`

这是最核心的实体，描述一个“被平台管理的 MCP 服务”。

它包含：

- 服务名称
- 传输方式（`stdio` / `sse` / `streamable_http`）
- 命令或 URL
- Bearer Token / 自定义 Header
- 会话模式
- 当前状态
- 错误次数与最后错误
- 标签

### 10.2 `Tool`

表示某个 MCP 服务同步回来的工具元数据。

### 10.3 `RequestHistory`

表示一次工具调用历史。

### 10.4 `AuditLog`

表示“谁在什么时候做了什么操作”的审计记录。

### 10.5 关系图

```text
MCPService 1 ---- n Tool
MCPService 1 ---- n RequestHistory
User       1 ---- n RequestHistory
User       1 ---- n AuditLog
```

## 11. 新手最值得重点看的 3 个 service

### 11.1 `service/mcp_service.go`

这是“服务管理主线”。

你能在这里看到：

- 创建服务
- 更新服务
- 删除服务
- 连接服务
- 断开服务
- 聚合状态
- 错误记录与告警

### 11.2 `service/tool_service.go`

这是“工具目录主线”。

负责同步某个服务的工具列表，并把工具信息保存下来。

### 11.3 `service/tool_invoke_service.go`

这是“工具调用主线”。

负责：

- 同步调用
- 异步调用
- 限流 / 并发控制
- 历史记录
- 任务查询 / 取消

如果你想理解“这个平台为什么不是简单 CRUD”，这里是重点。

## 12. `mcpclient` 这一层到底在干嘛

很多新手第一次看这里会有点懵。

你可以先只记住一句话：

```text
repository 管数据库里的“静态配置”
mcpclient 管进程内的“活连接和真实调用”
```

### `internal/mcpclient/manager.go` 的角色

它维护一个内存 map：

```text
serviceID -> managedClient
```

然后提供：

- `Connect`
- `Disconnect`
- `GetStatus`
- `ListTools`
- `CallTool`
- `Ping`

所以它更像一个：

```text
“运行中的 MCP 客户端连接池 / 会话管理器”
```

## 16. 新手最容易踩的坑

### 坑 1：以为数据库状态就是实时运行状态

不是。

项目里会同时存在：

- 持久化状态（数据库）
- 运行时状态（内存）
- 快照状态（Redis / snapshot）

所以 `service/mcp_service.go::Status` 才会做“多来源状态聚合”。

### 坑 2：不理解为什么有本地 runtime 和远程 runtime

因为项目支持双角色部署。

- `all`：本地直接调用 runtime
- `control-plane`：通过 RPC 调 executor

### 坑 3：只看 handler，不看 service

这个项目的业务规则主要在 `service`，不是在 handler。

handler 更多是在做：

- 绑定参数
- 取当前用户
- 调 service
- 返回统一响应

### 坑 4：忽略测试装配代码

`tests/testutil/app.go` 很值得看，它会直接告诉你作者期望这个系统如何被启动和验证。

## 19. 给初学者的最后建议

第一次看这个仓库时，不要试图一次性理解全部细节。

你只要先搞清楚这 4 件事：

```text
1. 服务怎么启动
2. 路由怎么进到 service
3. service 怎么分别操作数据库和 runtime
4. 测试是怎么把应用跑起来的
```

理解这四点后，后面的审计、异步任务、双角色 RPC、健康检查，都会变得顺很多。
